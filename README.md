# model-code-template

Template repository for model code, data, and results that can be used when publishing code for manuscripts or reports.

## Purpose

This repository provides a standardized structure for sharing scientific modeling code alongside journal publications or technical reports. The goal is to make analyses fully reproducible: a reader should be able to clone this repository, follow the setup instructions, run the scripts in order, and recreate all results, figures, and tables in the manuscript.

## Repository Structure

```
model-code-template/
├── README.md           # Project overview, setup, and reproduction instructions
├── LICENSE             # License governing code reuse
├── data/               # Input data files and data documentation
    ├── raw/            # Raw data files
    ├── processed/      # Processed data files
    └── data.md         # Description of data files
├── scripts/            # All analysis and modeling code
│   └── scripts.md      # Description of script execution order and purpose
└── results/            # All outputs generated by scripts (never edit by hand)
    ├── figures/        # Publication-ready figures
    ├── model-fits/     # Saved model objects and posterior samples
    └── tables/         # Summary tables
```

## How to Reproduce Results

Run scripts in the order documented in `scripts/scripts.md`. A complete analysis can typically be reproduced by running:

```
01_data_prep.R      # Prepare and clean input data
02_fit_model.R      # Fit the model
03_diagnostics.R    # Check convergence and model fit
04_results.R        # Generate figures and tables
```

All outputs will be written to `results/`. Do not manually edit files in `results/` — all outputs should be fully regenerated by the scripts.

## Reproducibility Best Practices

- **Set a random seed** in every script that uses stochastic methods (sampling, simulation, random initialization). Document the seed value.
- **Pin package versions** using `renv` (R) or `requirements.txt` / `conda environment.yml` (Python). Check in the lockfile so others restore identical environments.
- **Use relative paths only.** Never hardcode absolute paths (e.g., `/Users/yourname/...`). All paths should be relative to the project root.
- **Do not include credentials or API keys.** Use environment variables or a `.env` file listed in `.gitignore`.
- **Do not commit large data files to git.** Store them in a repository-adjacent location (e.g., a shared drive or data repository) and document access in `data/data.md`.
- **Document the full software environment** by saving `sessionInfo()` (R) or `pip freeze` (Python) output to `scripts/session_info.txt` at the end of a complete run.
